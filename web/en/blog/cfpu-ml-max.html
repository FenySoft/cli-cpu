CFPU-ML-Max

How We Optimized an ML Inference Chip
April 19, 2026 · Hocza József Szabolcs

The starting point

The CFPU many-core architecture is an obvious candidate for ML inference: thousands of Matrix Cores, each with a 4×4 INT8 MAC array, mailbox-based messaging, streaming dataflow. On paper: 328 TOPS (7nm, 500 MHz, ~20,500 cores). But analysis showed that the network bottleneck would limit sustained throughput to ~15% MAC utilization.

The problem was not the MAC array — it was the data feeding. The Turbo router's 42-bit wormhole link delivered ~5.2 bytes/cycle. The MAC consumed 32 bytes/cycle. A six-fold bottleneck. The MAC sat idle 85% of the time, waiting for data.

What we needed was to rethink the entire communication layer — not slow down the MAC, but speed up the network.

Step 1: Cell size reduction

The original interconnect cell payload was 128 bytes. This makes sense for the general-purpose CFPU: actor messages, serialized objects, GC data. But a Matrix Core's 4×4 tile is just 32 bytes of input — the cell was four times oversized.

Switching to a 64-byte cell reduced latency by 39% and router area by 24%. The power-of-two size was preserved, alignment stays simple. This doesn't sound revolutionary, but the freed area and wire budget became the foundation for the next steps.

Step 2: Systolic Wide router

This was the key insight: in ML inference, data flows in one direction. Activations from west to east, weights from north to south. There is no need for XY routing, VOQ, or iSLIP — these are general-purpose networking primitives that are unnecessary luxury for a Matrix Core.

We replaced the Turbo router with two 128-bit unidirectional links: W→E activation, N→S weight. The router shrank from 42,000 GE (Turbo) to ~5,000 GE — an 81% reduction. Its area: 0.001 mm².

But the size reduction is a side effect. The real win: the 128 bit/cc link delivers 16 bytes/cc in each direction. The MAC's 32 bytes/cc consumption: 16 bytes activation + 16 bytes weight. The link delivers exactly as much data as the MAC can consume.

With the Turbo router, the MAC sat idle 85% of the time. With the Systolic Wide router, it runs at 90–95%.

Step 3: Wider link — reallocating the wire budget

Where did the 128-bit link come from? The Turbo router's 42-bit link was bidirectional: N, S, E, W + local — 5 ports, each 42 bits, bidirectional. That is ~340 wires/core. The Systolic Wide router uses two 128-bit unidirectional links plus ~10 control wires: ~274 wires/core — fewer wires, more bandwidth.

The trick: removing XY routing, VOQ, and iSLIP saves not just logic but wires. No more address decoding, no arbitration signaling, no multi-port crossbar. The freed wire budget goes to the two main data paths.

Step 4: Post-MAC pipeline

Previously, MAC output was sent raw to the network: 16 INT32 values per tile = 64 bytes. But most ML layers are followed by ReLU, quantization, and pooling anyway. Why not do this in hardware, at the MAC output?

We integrated three units:

  • ReLU — a single comparator, 160 GE. If the value is negative, zero it.
  • Quantize INT32→INT8 — 300 GE. Quantizes the 32-bit MAC result to 8 bits.
  • Max-Pool 2×2 — 200 GE. Selects the maximum from four values.

Total: 660 GE (~0.0001 mm²). Output traffic: 64 bytes → 4 bytes. 16x reduction.

This is not just about offloading the network — the freed output link bandwidth becomes available for more input data. The post-MAC pipeline delivers dramatic network efficiency for effectively zero area cost.

Step 5: SRAM sweet spot reversal

The earlier analysis recommended 32 KB SRAM as optimal: enough for weight caching, with weight reuse compensating for lost cores. But the Systolic Wide router reversed this logic.

If the link bandwidth alone is sufficient to feed the MAC (128 bit/cc = 16 bytes/cc), then the weight cache advantage is minimal. Larger SRAM takes more area, fewer cores fit, and sustained TOPS drops — despite barely improving MAC utilization.

8 KB SRAM became the new optimum: enough for code, stack, and MAC buffer, and it enables +18% more cores compared to the 32 KB variant.

Step 6: Clock optimization

The final step was choosing the clock frequency. TOPS scales linearly with clock, but power scales supra-linearly. Analysis:

Clock TOPS (609 mm²) TDP (est.) TOPS/W
500 MHz ~512 ~70 W ~7.3
1 GHz ~1,024 ~210 W ~4.9
2 GHz ~2,048 ~800 W ~2.6

1 GHz is the best compromise: 2x TOPS over 500 MHz, at still-acceptable TOPS/W. 2 GHz is not worth it — power goes up 4x while TOPS only doubles.

⚠ Arithmetic projections — CFPU-ML-Max figures are pre-synthesis, pre-RTL estimates. They indicate design directions, not measured performance data.

Final result: CFPU-ML-Max vs NVIDIA (projected)

After the five optimization steps, CFPU-ML-Max achieves 90–95% sustained MAC utilization — up from ~15%. How does this look compared to NVIDIA?

The table below compares at the same die size, same process node (~5nm/4nm), dense INT8 (without sparsity):

Comparison CFPU-ML-Max (sust.) NVIDIA (sust.) Projected advantage Projected TOPS/W
vs RTX 4060 (159 mm²) 242–256 TOPS 70–122 TOPS 2.0–3.7x 4–7x
vs RTX 4090 (609 mm²) 922–973 TOPS 264–462 TOPS 2.0–3.7x 4–7x
vs H100 (814 mm²) 1,233–1,301 TOPS 792–1,385 TOPS 0.9–1.6x 2–4x

The numbers are honest. For RTX 4060 and 4090, CFPU-ML-Max delivers 2–3.7x better sustained TOPS because only ~30–35% of the GPU die is ML hardware. For H100, the picture is more balanced: H100 is almost entirely ML-dedicated with HBM, while CFPU-ML-Max is SRAM-only.

Why is NVIDIA sustained low?

1. Die area sharing — the RTX 4060/4090 die is ~30–35% Tensor Cores. The rest is rasterization, RT cores, video, display engine. CFPU-ML-Max: 100% ML.
2. GDDR bottleneck — Tensor Cores consume data faster than GDDR6X can deliver.
3. Non-ML hardware consumes power — GPU TDP includes rasterization, video decoder, display engine.

CFPU-ML-Max is not an LLM accelerator. It is optimized for small and medium inference models (MobileNet, ResNet, YOLOv8, BERT-Base) where the entire model fits in distributed on-chip SRAM. LLaMA-7B (7 GB) does not fit — that would require a DRAM controller, which is not part of the current architecture.

TDP estimates are pre-synthesis. The TOPS/W advantage is promising, but validated power figures require at minimum RTL-level power analysis.
