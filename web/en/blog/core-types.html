Five Cores, One Chip

The CFPU core family: Nano, Actor, Rich, Matrix, Seal
April 18, 2026 · Hocza József Szabolcs

Dual specialization

The CFPU is not a single core multiplied by thousands. Different workloads demand different processor geometries — which is why the CFPU core family branches in two directions from a single common base: the Nano Core.

The first branch grows toward actor processing: Nano → Actor → Rich. Each step adds more SRAM, more registers, and a richer ISA — Actor is optimized for general-purpose CIL execution, Rich for floating-point computation and GC-managed object handling.

The second branch diverges toward ML/SNN compute: Nano → Matrix. The Matrix Core is not a scaled-down Actor or Rich — it is a direct descendant of Nano, extended with MAC+SFU units. Small, massively parallel, streamable. 

Nano Core (base)
  ├── Actor Core  (general actor, 64 KB SRAM)
  │     └── Rich Core   (FP + GC + Obj, 256 KB SRAM)
  └── Matrix Core (ML/SNN compute, 8 KB SRAM + MAC+SFU)

Seal Core (security core, independent lineage)

This dual branching is deliberate: the Rich Core is expensive (256 KB SRAM, GC, FPU), while the Matrix Core's goal is to maximize the number of cores per die. Direct descent from Nano keeps the area footprint low.

The five core types

Nano Core — spike / sensor

The smallest and most numerous core. Integer arithmetic only, 4 KB SRAM, minimal ISA, mailbox-based messaging. Designed for sensors, edge processing, and SNN spike propagation — where precision is less important than speed and energy efficiency.

Actor Core — general actor

The CFPU's primary workhorse. Full CIL ISA with GC and object support, 64 KB SRAM, mailbox-based messaging. Neuron OS actors, server logic, general-purpose parallel processing. In most CFPU configurations, this is the dominant core type.

Rich Core — FP actor

An extension of the Actor Core with a floating-point unit (FPU). 256 KB SRAM. Numerical computation, scientific workloads, complex business logic. The baseline for all external CPU comparisons.

Matrix Core — ML/SNN compute

A Nano descendant extended with MAC+SFU (Multiply-Accumulate + Special Function Unit). 8 KB SRAM, streaming dataflow. Optimized for matrix multiplication, convolution, and Spiking Neural Network propagation. Massively parallel: at 7nm, ~20,500 cores fit on a single 800 mm² die (including Turbo router and infrastructure overhead).

Seal Core — authentication

A security core placed at the center of the chip, integrated with the L3 crossbar. All cross-region traffic naturally passes through it. Code authentication, key management, isolation boundary enforcement. Not a compute core — a security gatekeeper.

Matrix Core: Nano on steroids

The first question that arises: why not derive the Matrix Core from the Rich Core? Rich has an FPU, large SRAM — isn't that exactly what ML needs?

The answer is area efficiency. A Rich Core with 256 KB SRAM and GC is ~107,000 gates. In the same silicon area, you can fit three times as many Matrix Cores — and ML/SNN workloads depend not on the speed of one large core, but on the number of parallel MAC operations.

The Matrix Core's 8 KB SRAM is not a limitation — it is the streaming sweet spot. A convolutional layer or SNN propagation pass typically fits in a small local buffer, then passes the result forward. 8 KB is exactly enough to process one tile. More SRAM would be waste.

Matrix Core is not Rich Core minus the FPU. It is Nano Core plus MAC+SFU — and that difference means 3× more cores on the same die.

The MAC+SFU unit (multiply-accumulate + sigmoid/tanh/ReLU hardware) slots directly into the CIL pipeline. Keeping the integer Nano ISA ensures that control logic (loops, branches, message passing) remains equally efficient.

CFPU-ML: the ML/SNN chip

CFPU-ML configuration (7nm, 800 mm², estimated):

~19,500 Matrix Cores — MAC+SFU streaming array (8 KB SRAM, Turbo router)
~1,000 Actor Cores — supervisor, scheduler, coordination (64 KB SRAM)
1 Seal Core — model authentication, isolation

Peak throughput: ~312 INT8 TOPS* — theoretical maximum, every MAC busy every cycle
Sustained throughput: ~49–176 INT8 TOPS* — realistic range depending on optimization strategy

Key advantage: Every Matrix Core is an independent actor — no shared memory, no driver stack, directly programmable from Neuron OS via CIL messages.

* Peak = core count × MAC units × clock. Sustained range depends on SRAM sizing and data-feeding strategy (weight reuse, systolic chaining, post-MAC processing). The MAC array is ~6× faster than the network can feed it — sustained throughput is network-limited, not compute-limited. No RTL simulation or silicon measurement exists yet.

The CFPU-ML is not a GPU competitor — it is a different paradigm. Where a GPU is one massive, monolithic shader array, the CFPU-ML is ~20,000 independent ML actors communicating via messages. Partial failures leave healthy cores running; there is no global memory, no pointer manipulation, no CUDA/ROCm driver stack.

Why CFPU-ML?

The CFPU-ML's value lies not in raw TOPS but in a combination of architectural properties that no existing chip offers together: programmable neuron model (arbitrary CIL), event-driven operation (0W idle), isolated cores without shared memory, open-source hardware (CERN-OHL-S), and native .NET CIL execution.

The CFPU-ML-Max variant — with Systolic Wide router, post-MAC pipeline, and 1 GHz clock — achieves 90–95% sustained MAC utilization and delivers 2–3.7× better sustained TOPS than NVIDIA RTX 4060/4090 at the same die size.
